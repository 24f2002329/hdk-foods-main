 HDK Foods — Production Readiness Plan

 Context

 Payment flow is working end-to-end (Cashfree sandbox, retries, reconciliation).
 Next goal is to
 harden the backend for real users, close security holes discovered in the code

 Phase 1 — Backend Security & Hardening

 1a. Fix the open OrderListView

 File: backend/orders/views.py + backend/orders/urls.py

 OrderListView has no permission_classes, exposing every order in the system
 publicly.
 - Add permission_classes = [IsAdmin] to OrderListView.
 - Add IsAuthenticated import already exists; use IsAdmin from
 authentication.permissions.

 1b. Fix unhandled 500s on missing orders

 File: backend/orders/views.py

 ConfirmOrderView, RejectOrderView, UpdateOrderStatusView, AssignDeliveryView
 all use
 Order.objects.get(pk=pk) without try/except. Wrap each in try/except
 Order.DoesNotExist and
 return HTTP 404 — matching the pattern already used in SelectPaymentView.

 1c. Validate status choices in UpdateOrderStatusView

 File: backend/orders/serializers.py + backend/orders/views.py

 UpdateStatusSerializer accepts any string. Change status field to a ChoiceField
 with
 STATUS_CHOICES from the Order model, or define the valid list inline in the
 serializer.

 1d. Populate confirmed_by on ConfirmOrderView

 File: backend/orders/views.py

 ConfirmOrderView never sets order.confirmed_by. Add order.confirmed_by =
 request.user and
 include "confirmed_by" in the update_fields list.

 1e. Restrict CORS and ALLOWED_HOSTS

 File: backend/config/settings.py

 - Change ALLOWED_HOSTS = ["*"] to read from env: ALLOWED_HOSTS =
 os.getenv("ALLOWED_HOSTS", "localhost,127.0.0.1").split(",")
 - Change CORS_ALLOW_ALL_ORIGINS = True to CORS_ALLOW_ALL_ORIGINS =
 os.getenv("CORS_ALLOW_ALL_ORIGINS", "False") == "True" and add
 CORS_ALLOWED_ORIGINS = os.getenv("CORS_ALLOWED_ORIGINS", "").split(",") for
 production use.
 - Add DEBUG = os.getenv("DEBUG", "False") == "True" (currently hardcoded True).
 - Update backend/.env with DEBUG=True, ALLOWED_HOSTS=*,
 CORS_ALLOW_ALL_ORIGINS=True so dev behaviour is unchanged.

 1f. Add structured logging

 File: backend/config/settings.py + backend/orders/views.py

 Add a LOGGING dict to settings — console handler in dev, can redirect to
 file/service in prod.
 Add logger = logging.getLogger(__name__) to orders/views.py and log:
 - Payment creation (order id, amount, environment)
 - Verification calls (Cashfree status received)
 - Webhook events (see 1g)
 - Any unexpected gateway responses

 1g. Cashfree Webhook endpoint

 Files:
 - backend/orders/views.py — new CashfreeWebhookView
 - backend/orders/urls.py — new path "webhook/cashfree/"
 - backend/config/settings.py — add CASHFREE_WEBHOOK_SECRET =
 os.getenv("CASHFREE_WEBHOOK_SECRET", "")
 - backend/.env — add CASHFREE_WEBHOOK_SECRET= (fill in from Cashfree dashboard)

 Cashfree sends a POST to the webhook URL with a JSON body and an
 x-webhook-signature header
 (HMAC-SHA256 of the raw body using the webhook secret). The endpoint must:
 1. Be exempt from CSRF (@csrf_exempt via dispatch) and use AllowAny permission
 (no JWT).
 2. Verify the signature: hmac.new(secret, raw_body, sha256).hexdigest() ==
 header value.
 3. Parse event_type from body. On PAYMENT_SUCCESS_WEBHOOK: look up order by
 cashfree_order_id
 from data.order.order_id, set payment_status="paid",
 payment_id=data.payment.cf_payment_id.
 4. Return HTTP 200 immediately (Cashfree retries on non-200). Return 400 only
 for bad signature.
 5. Log every event (type + order id).

 This is the production-grade fix for the "payment succeeded but status stayed
 pending" class of
 bugs — it doesn't rely on the client polling or calling verify at all.
